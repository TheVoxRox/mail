# Windows code signing + Tauri updater

Signed Windows releases are built by the `Windows Signed Release` GitHub Actions
workflow. It checks out the monorepo, packages the backend sidecar, imports a
PFX code-signing certificate into the runner certificate store, signs the
sidecar executable, builds the Tauri bundle with Windows signing and updater
artifacts enabled, generates `latest.json`, verifies the resulting signatures,
and uploads the signed bundle artifacts to the target GitHub Release.

## Required GitHub secrets

- `WINDOWS_CERTIFICATE_BASE64`: base64-encoded `.pfx` code-signing certificate.
- `WINDOWS_CERTIFICATE_PASSWORD`: export password for the `.pfx` certificate.
- `WINDOWS_CERTIFICATE_THUMBPRINT`: SHA-1 thumbprint of the signing certificate.
- `TAURI_UPDATER_PUBKEY`: public key generated by the Tauri signer.
- `TAURI_SIGNING_PRIVATE_KEY`: private key generated by the Tauri signer. Keep it secret.
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`: optional private-key password if one was set.
- `GOOGLE_OAUTH_CLIENT_ID`: Google "Desktop app" OAuth client ID (public identifier).
- `GOOGLE_OAUTH_CLIENT_SECRET`: Google "Desktop app" client secret. Non-confidential per Google's installed-app guidance — it is shipped in the build and the flow is protected by PKCE. Without it Google login does not work in the release.
- `MICROSOFT_OAUTH_CLIENT_ID`: Microsoft Entra application (client) ID. Microsoft is a public client, so no client secret is required or shipped.

## Optional repository variables

- `WINDOWS_DIGEST_ALGORITHM`: defaults to `sha256`.
- `WINDOWS_TIMESTAMP_URL`: defaults to `http://timestamp.digicert.com`; prefer
  the timestamp URL recommended by the certificate issuer.
- `TAURI_UPDATER_ENDPOINTS`: update endpoint list. Use either a comma-separated
  list, a newline-separated list, or a JSON array. Production endpoints must
  use HTTPS. A static GitHub release JSON URL works, for example
  `https://github.com/<owner>/<repo>/releases/latest/download/latest.json`.
- `TAURI_UPDATER_WINDOWS_INSTALL_MODE`: defaults to `passive`.

## Creating the base64 certificate payload

On Windows:

```powershell
[Convert]::ToBase64String([IO.File]::ReadAllBytes("certificate.pfx")) |
  Set-Content -NoNewline windows-certificate.base64.txt
```

## Local signed build

Import the `.pfx` into `Cert:\CurrentUser\My`, set `WINDOWS_CERTIFICATE_THUMBPRINT`,
`TAURI_UPDATER_PUBKEY`, `TAURI_UPDATER_ENDPOINTS`, and `TAURI_SIGNING_PRIVATE_KEY`,
optionally set `WINDOWS_DIGEST_ALGORITHM`, `WINDOWS_TIMESTAMP_URL`, and
`TAURI_SIGNING_PRIVATE_KEY_PASSWORD`, then:

```powershell
npm run sidecar:sync:windows
npm run tauri:release-config:windows
.\scripts\sign-windows-file.ps1 -Path .\src-tauri\binaries\mail-x86_64-pc-windows-msvc.exe
npm run tauri:build -- --config src-tauri/tauri.release.conf.json
.\scripts\verify-windows-signatures.ps1
```

## Generating Tauri updater keys

```powershell
npm run tauri -- signer generate -- --write-keys .tauri\updater.key
```

The command prints the public key for `TAURI_UPDATER_PUBKEY`; store the private
key content in `TAURI_SIGNING_PRIVATE_KEY`.

## Release artifacts

The release build produces updater artifacts and `.sig` files beside the
platform bundles. The workflow publishes:

- `voxrox-mail-<version>-windows-x64-setup.exe` (normalized name)
- `voxrox-mail-<version>-windows-x64-setup.exe.sig` (Tauri updater signature)
- `voxrox-mail-<version>-windows-x64-setup.exe.sha512` (integrity checksum)
- `latest.json` with `windows-x86_64.signature` set to the generated `.sig`
  contents

The workflow also records a Sigstore-backed **build-provenance attestation** for
the installer (`actions/attest-build-provenance`). It ties the binary to the
workflow + commit that produced it and is the strongest free authenticity check
for a build without a paid Authenticode certificate. Verify a downloaded
installer with `gh attestation verify <file> --repo <owner>/<repo>`. The bare
`.sha512` only guards against transport corruption — it shares the release
channel with the installer, so it is not a tamper defense on its own.

Windows downgrades are disabled; release rollback must follow the operational
recovery procedure in [`../../backend/OPERATIONS.md`](../../backend/OPERATIONS.md)
instead of installing an older bundle over a newer one.

## Generated config files (gitignored)

`src-tauri/tauri.release.conf.json`, `src-tauri/tauri.signing.conf.json`, and
`src-tauri/tauri.updater.conf.json` are generated from environment variables and
ignored by Git so release metadata is not committed accidentally.
