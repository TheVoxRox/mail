# Changelog

Release-level changelog pro celé monorepo VoxRox Mail.

Backend a frontend mají vlastní verze, ale desktop release se vydává jako jeden
Tauri bundle se sidecar backendem. Každý release proto drží jednu společnou
sekci s podsekcemi podle artefaktu.

## Unreleased

### Backend

- Přidán sidecar lifecycle kontrakt: graceful shutdown, `session.json` + `.ready` gate.
- Přidán first-run crypto bootstrap do `${app.data-dir}/crypto.bin`; `MAIL_CRYPTO_KEY/SALT` zůstávají volitelný dev override.
- Konsolidace startup handshake do `/api/v1/system/readiness` (vrací `appVersion`, `apiVersion`, `minClientVersion`, `dbSchemaVersion` a `phase`). Původní `/api/handshake` endpoint odstraněn — desktop klient používá readiness.
- Pre-migration DB backup běží jen když má Flyway pending migrace (před každým release majoru jeden snapshot, ne při každém startu).
- Health gate při startu: při selhání `PRAGMA quick_check` se aplikace zastaví s recovery zprávou odkazující na `OPERATIONS.md`.
- Audit event `app_started` s `appVersion`, `dbSchemaVersion`, `previousAppVersion`.
- Spring AOT a JEP 483 AOT cache (opt-in přes packaging script): cold start jvm `java -jar` 8,5 s → 5,4 s (−37 %).
- Hybrid pagination fáze 1: `MailFacade.getEmails` má rychlý lokální fast-path + lazy IMAP fetch jen pokud klient žádá novější stránku. `FolderCountCache` drží server-side count per `(account, folder)` s TTL 60 s — eliminuje opakované `STATUS` volání na IMAP při scrollování.
- Send notifikace přes SSE — odeslání e-mailu vrací `sendId` v 202 odpovědi, async send výsledek doručen jako `send_completed` / `send_failed` event přes `SseNotificationService`. Frontend tak ukazuje pravdivý stav „odesílá se…" → „odesláno" / „odeslání selhalo" místo optimistického toastu.
- IMAP fetch robustness: pokud server vrátí malformed BODYSTRUCTURE pro konkrétní zprávu (typicky Seznam), `MessageFetcher` nově persistuje envelope-only stub místo aby celou zprávu zahodil. Uživatel zprávu uvidí v listu se subject/sender/date, content fetch se zopakuje při otevření detailu.
- Bezpečnost (audit B1-1): tělo zprávy se z IMAP nově čte s limitem 8 MB (bounded stream, `partialfetch` explicitně připnutý) místo neomezeného načtení do paměti — nepřátelský server už nemůže obřím tělem shodit sidecar. Příliš velká zpráva dostane persistentní příznak `body_oversize`, detail vrací lokalizovaný placeholder „zpráva je příliš velká" a opakované otevření už na server nesahá; u `multipart/alternative` se místo placeholderu použije textová alternativa (bez ohledu na pořadí částí), pokud se do limitu vejde. Odpověď/přeposlání takové zprávy cituje prázdné tělo, nikdy text placeholderu. Neznámá znaková sada těla nově degraduje na UTF-8 místo chyby načtení.
- Oprava zobrazení: `multipart/alternative` s vnořeným `multipart/related` (typicky Apple Mail — HTML s vloženými obrázky) se nově vykreslí jako HTML; dřív se tichým fallbackem zobrazila jen textová verze.
- Hostile-content IMAP harness (`MailContentGreenMailIT`): limit těla 8 MB, výběr alternativ i charset fallbacky ověřeny end-to-end přes živý in-process IMAP server produkčním klientským stackem — dynamický doplněk statického auditu B1.
- Log hygiene re-audit: čtyři interní texty výjimek (duplicitní účet/kontakt/e-mail) nesly nemaskovaný e-mail a tekly do `mail.log` — nově maskovány u zdroje; odpověď v UI beze změny. Kompletní záznam re-auditu v `backend/SECURITY_RELEASE_CHECK.md`.
- Oprava boot blockeru: backend padal při startu na nerozresitelném placeholderu `microsoft.client-secret` (Microsoft je public client, property už neexistuje). `client_secret` přesunut ze sdíleného OAuth refresh body do Google-specific hooku; Microsoft cesta secret vůbec neposílá (public client by ho odmítl s AADSTS700025). Navíc po selhání startu JVM nově hned skončí s exit 1 — dříve ji ne-démonová vlákna držela naživu jako zombie proces, který blokoval data dir a další start končil exit 78 „already running".
- Programmer text v Java zdrojích, Flyway migracích a packaging skriptech přeložen do angličtiny. Strict lint v CI gate.
- Bezpečnost odesílací cesty (review odesílání, batch 1): `POST /drafts/{stableId}/send` nově synchronně validuje, že stableId je vlastní koncept účtu ve složce Drafts (`MailFacade.verifyDraftForSend` byl napsaný a otestovaný, ale nikdy nevolaný) — bez toho async cesta re-odešle a expunge-ne libovolnou zprávu, na kterou stableId ukazuje (chybné id = re-odeslání přijaté zprávy z INBOXu původním příjemcům a její trvalé smazání). Stejná díra u `saveDraftAsync ?replaces=`: cíl se hard-delete-ne jen pokud je to vlastní zpráva účtu ve složce Drafts (jinak se ponechá), takže špatné `replaces` id už neexpunge cizí ani přijatou poštu.
- Pravdivost výsledku odeslání (review odesílání, batch 1): selhání post-send úklidu (zápis do DB / append do Sent) po úspěšném `transport.sendMessage` už nehlásí `send_failed` — dřív SQLITE_BUSY po úspěšném odeslání vedl k falešnému „odeslání selhalo" a uživatel poslal zprávu podruhé (duplicitní doručení; u konceptů horší, protože draft už byl smazán). Klientův pending stav se navíc rozřeší na faktu doručení. A `recordSendFailure` nově broadcastuje `send_failed` i když zápis chyby do DB selže — dřív DB chyba při zaznamenání selhání nechala odesílání „odesílá se…" navždy.
- Robustnost odesílací cesty (review odesílání, batch 2): `MimeMessageBuilder` odmítá CR/LF v polích, která jdou verbatim do MIME hlaviček (subject, In-Reply-To, References, attachment Content-Type) — defense-in-depth proti header injection (ohraničené B3, reply cesta byla bezpečná díky IMAP unfoldingu, ale klient tato pole nesanitizoval). `sendDraftAsync` čte `draftUid`/folder uvnitř try — null UID u poškozeného řádku dřív hodil NPE mimo try a nechal klienta viset bez `send_failed`. Opraven i zavádějící javadoc `ImapAppendService` („reconciled by next sync" neplatí pro selhaný Sent append na serverech bez auto-file).
- Identita konceptu vzniká při uložení (draft lifecycle, PR-A): `POST /drafts` nově vrací v 202 deterministické `stableId` — Message-ID se konceptu přiděluje před dispatch-em async appendu (dřív vznikal implicitně až při `writeTo`, pozdě na vrácení klientovi) a `stableId` je jeho hash. Append jde přes UIDPLUS `APPENDUID` a řádek zprávy se upsertne lokálně hned po úspěšném appendu (bez UIDPLUS se jako dřív objeví až se syncem), takže `?replaces=` řetězení a okamžité znovuotevření konceptu fungují bez čekání na sync. Klientská polling heuristika (list + match podle předmětu) tím ztrácí důvod existence — odstraní se v PR-B.
- Odeslání vlastní úklid konceptu (draft lifecycle, PR-A): `POST /messages/account/{id}/send?supersedesDraftId=` — koncept, ze kterého zpráva vznikla, maže backend až PO doručení (se stejným ownership/folder guardem jako `replaces`); klient ho už nemá mazat po 202, protože async selhání by nechalo obsah bez jediné kopie. Při selhání odeslání nové zprávy (bez supersede) se obsah best-effort uloží jako recovery koncept do Rozepsaných a `send_failed` notifikace nese jeho `recoveryDraftStableId`.
- Bcc přežije znovuotevření konceptu (draft lifecycle, PR-A): nový sloupec `recipients_bcc` (V1 in-place, mimo FTS index), `MessageFetcher` čte `RecipientType.BCC`, `MailDetailResponse.recipientsBcc` a compose prefill ho vrací do pole Skrytá kopie — dřív se Bcc při reopenu tiše zahodilo.
- Oprava „nesmrtelných" zpráv v koši: `DELETE /messages/{stableId}` na zprávu, která už v koši je, dřív dispatchnul přesun koš→koš — na serveru no-op, takže další sync lokálně smazané zprávy vzkřísil (ověřeno v logu: 101× „Moving UID x from [Gmail]/Koš to [Gmail]/Koš"). `MailFacade.moveToTrash` nově detekuje zprávu už v trash složce a místo přesunu ji trvale smaže (`ImapActionService.hardDeleteAsync` — `\Deleted` + EXPUNGE, audit `mail_purge`/`imap_purge`, metrika `mail.imap.purge`). OpenAPI popis endpointu aktualizován.

- Statická analýza: NullAway 0.13.7 (z 0.13.6) nově hlásí init warningy i pro JPA konstruktory s argumenty. Pole spravovaná Hibernate (`@Id`, `@GeneratedValue`, `@Version`) jsou vyjmuta přes `ExcludedFieldAnnotations` — jsou null jen v transientním okně před `save()`. Dva skutečně nullable sloupce `folder_sync_state` (`uid_validity`, `last_sync_at`) jsou nově poctivě `@Nullable`: kód je tak už dávno četl (`FlagSyncService.handleUidValidity` na ně testuje null), jen to typ nepřiznával. `MessageDownloader` invariant „UIDVALIDITY je vyřešená před stahováním" nově explicitně ověřuje místo aby null tiše protekl do `NOT NULL` sloupce `messages.uid_validity` a do `MessageStableId`.

- Uzavřen nález „Gmail Sent duplicita" z review odesílací cesty: append do Sent zůstává vědomě bez provider gatingu. Ověřeno živě proti reálnému Gmail účtu (2026-07-14) — přímý send i draft-send nechají v Odeslané právě jednu kopii, protože Gmail náš append sloučí se svou vlastní kopií podle Message-ID; v logu přitom není žádné selhání appendu (to by hlásilo WARN). Gating by hlídal neexistující problém. Zjištění i podmínky pro přehodnocení zapsány do javadocu `ImapAppendService`.

- Detail zprávy nese složku: `MailDetailResponse.folderName` (nález z review #165) — klient bez něj nedokázal u otevřené zprávy rozlišit, že leží v koši, a mazání z detailu mohlo obejít potvrzení trvalého smazání (detail otevřený + seznam odstránkovaný jinam). Interní produceri DTO (sync fetch, draft upsert) složku znají a předávají ji; `toEntity` ji dál bere jako parametr, pole je čistě informační pro API. OpenAPI snapshot + `schema.d.ts` přegenerovány.

- Koncept s rozepsanou adresou se nově uloží (nález z živého testu 2026-07-14): `MimeMessageBuilder` dostal explicitní `AddressPolicy` — odeslání parsuje adresy striktně jako dřív (STRICT: tichý výpadek příjemce nesmí nastat), ale draft-save (DRAFT) nedokončený token (`luke.lacina@`) z hlavičky vynechá, místo aby celé uložení spadlo s `AddressException` (ERROR log, `AuditLog.failure`, probliknutí chyby v UI — a při zavření compose ztráta textu, protože se neuložila žádná revize). Nedokončený token se záměrně nezapisuje na drát: malformovaný addr-spec rozbije IMAP ENVELOPE a takový koncept by se ze serveru už nedal přečíst (`MessageFetcher` zprávu s nečitelnou envelope přeskakuje); raw text pole přitom zůstává v lokálním řádku, takže composer ho dál zobrazuje. Ověřeno end-to-end v `DraftLifecycleGreenMailIT` (nová fáze 4: uložení s rozepsanou adresou + čitelný re-sync ze serveru).

- Odeslání konceptu (draft lifecycle, oprava po SR testu): supersede konceptu po úspěšném odeslání nově krátce retryuje lokální dohledání řádku (3× / ~800 ms). Souběžný draft-save (append + upsert lokálního řádku) běží na vlastním async tasku, takže hned po odeslání ještě nemusel řádek založit — supersede ho pak minul a koncept po odeslání zůstal v Rozepsaných (i na Gmailu s UIDPLUS). Zrcadlí retry, který už používá klientský discard.

### Backend — threading / konverzace (Phase 1, backend-only)

- V2 Flyway migrace `add_threading.sql` přidává `thread_id` / `thread_root_message_id` / `thread_position` sloupce na `messages` + 2 composite indexy.
- `ThreadingService` implementuje JWZ-light algoritmus: direct In-Reply-To, References walk oldest-to-newest (cap 50), nový thread fallback, late-arriving-parent reconciliation. Per-account scope. Skip subject clustering (false positives na běžných subjectech).
- `ThreadingBackfillService` běží na `ApplicationReadyEvent` async — backfillne `thread_id` pro každý null řádek active+!requires_reauth účtů s audit eventy.
- Nový endpoint `GET /api/v1/messages/account/{accountId}/threads/{threadId}` vrací `ThreadResponse` (membery v ascending `threadPosition` order, unreadCount).
- Nový hidden `POST /api/internal/threading/recompute?accountId=N` pro QA / support reset.
- SSE `oneOf` rozšířen o `thread_updated` event — emitován při novém append do existujícího threadu nebo merge orphan chain.
- `MailSummaryResponse` + `MailDetailResponse` advertizují `threadId` (nullable). V0.1.0 frontend nepoužívá — typy připravené pro V0.2 UI.

Otevření zprávy nově deklaruje záměr (`mail/bodyFocus.ts`) — vědomé otevření (Enter/Space, klik, přímý odkaz, výsledek hledání) přesune čtecí kurzor do těla, ale změna řádku, za kterou se jen táhne výběr do čtecího podokna, ho nechá na buňce gridu. Dřív si tělo bralo fokus při každé změně `stableId` a protože sedí v sandboxovaném iframu, další šipka se do seznamu už nedostala — uživatel po prvním ArrowDown uvízl ve čtecím podokně.
- Návrat fokusu na řádek po zavření zprávy: Esc i tlačítko/odkaz „Zpět do složky" nově vrací fokus na řádek, ze kterého se zpráva otevřela. Ve split režimu obnova doběhla dřív, než navigace na složku doskočila, a SvelteKit ji svým resetem fokusu přepsal na `<body>` → `<main>`; navigace při zavření detailu proto jde s `keepFocus`. Breadcrumb odkaz zpět v režimu bez podokna čtení šel dosud mimo tuto cestu úplně a fokus nevracel vůbec. Obnovený fokus navíc nikdy nekotví na zaškrtávátku výběru (po smazání nemá co říct) — stejně jako u sloupce akcí míří na buňku předmětu.
- Hledání drží uživatele ve výsledcích: Esc (i tlačítko Zpět) na detailu otevřeného výsledku zavře detail na místě a vrátí fokus na jeho řádek — dosud šel přes zavírací cestu Pošty a odnavigoval do naposledy prohlížené složky, takže výsledky i dotaz zmizely a fokus skončil na náhodném řádku doručené pošty. `MessageDetail` k tomu dostal prop `onClose`. Smazání/přesun výsledku přes řádkové menu nově taky vrací fokus na sousední řádek; obnova bydlí ve stránce hledání, protože reload výsledků projde stavem `loading`, který grid odmountuje.
